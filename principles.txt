To make your RSS.com cron service resilient against attacks, easy to maintain, and readable for other developers, you must adhere to these foundational software engineering principles.1. Cyber Security & System Defense PrinciplesThese principles protect your server against typical external vulnerabilities when parsing untrusted, public RSS fields:Server-Side Request Forgery (SSRF) Prevention: Never trust user-provided or external RSS feed URLs blindly. Before making an HTTP request, resolve the domain name to its IP address and match it against an IP Blocklist. Instantly drop connections targeting localhost (127.0.0.1), private subnets (192.168.0.0/16), or cloud metadata endpoints (169.254.169.254) to stop malicious actors from scanning your internal network infrastructure.XML External Entity (XXE) Injection Defense: RSS feeds are structured as XML documents. Malicious feeds can hide custom DOCTYPE or external entity declarations designed to extract configuration profiles or cause server crashes. Always pass configuration flags to your XML parser node that explicitly disable internal and external DTD (Document Type Definition) parsing.Input Sanitisation & Fail-Fast Structure Checking: Validate the schema of any data payload immediately at the boundary of your system using tools like Zod. If a fetched RSS feed does not match your exact TypeScript model, reject it at step 01-sanitize.ts before passing it to internal database logic.Data Injection Prevention via Parameterised Queries: Leverage your TypeScript ORM (Prisma or Drizzle) to build data transactions. This naturally creates parameterised database inputs, completely neutralizing any potential SQL injection vectors inside incoming RSS strings.2. Architectural & Structural PrinciplesThese concepts govern how code components interact, ensuring clear system boundaries and isolation:Separation of Concerns (SoC): Divide your code into distinct sections based on what they do. In this codebase, the Next.js API layer handles network ingress, the core handles time tracking, the service layer manages RSS network fetches, and the analysis folder processes text metrics. They do not overlap.Layered Architecture (Strict Directional Dependency): Arrange your code in a clear vertical stack (API/Cron Workers → Analysis Engine → Data Access Layer). Code inside your core layers should never import or know about files in your outer routing layouts.Process Isolation: Keep execution tasks completely independent. If an analysis worker crashes due to a corrupt RSS payload, it should only terminate that specific execution string without knocking down your public Next.js API routes or the core timing engines.3. Maintainability & Flexibility PrinciplesThese rules ensure your codebase can easily adapt to new feature updates, structural changes, or third-party service adjustments:Dependency Inversion Principle (DIP): Depend on abstractions rather than concrete choices. Your scheduler shouldn't care how a message is queued; it should call an internal IQueue interface. If you replace Redis with AWS SQS later, you only change your setup file inside src/lib/, leaving the core engine untouched.Open/Closed Principle (OCP): Your code should be open for extension but closed for modification. If you need to add a fifth analysis check to your workflow, you should create a new file in src/analysis/steps/05-new-rule.ts and add it to the pipeline sequence array instead of rewriting old steps.Single Responsibility Principle (SRP): Every function, class, or script file must do exactly one thing. A step function should only lookup data inside your static JSON dictionaries; it shouldn't also attempt to write logs to the central database or parse XML strings.4. Code Quality & Readability PrinciplesThese principles focus on code design, keeping it approachable for any developer joining your project:Structured, Traceable Logging: Do not use plain string logs (console.log). Output log statements natively as structured JSON entries (info(), error()) tagged with a unique running execution ID (traceId). This enables engineers to easily trace a failed run from the initial cron call right through the exact processing step.Intentional, Declarative Naming: Avoid short, cryptic labels. Name functions and variables explicitly after what they do (e.g., rssFeedProcessingPipeline instead of rProc).Why-Based Commenting: Write your source code like clear prose that reads naturally. Use comments exclusively to document business context, edge cases, or deliberate security defenses (e.g., “// XXE guard: DTD must remain disabled here to block unsafe payloads”) rather than detailing obvious programming operations.KISS (Keep It Simple, Stupid) & YAGNI (You Aren't Gonna Need It): Refrain from over-complicating workflows early on. Do not configure a heavy distributed database system or an intricate streaming broker until your target RSS workload explicitly demands it.
